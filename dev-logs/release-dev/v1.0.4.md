# v1.0.4 (July 14, 2026)

- Fixed parser handling for `||` so logical OR expressions no longer get misread as empty pipe delimiters in normal expressions and declarations.
- Fixed function-type checks so `fn` assignments and declarations accept function values consistently across the tree-walk interpreter and bytecode VM.
- Fixed WASM and CLI builtin registration for `@println` as it registered a `reference error` whenever called, this was due to how functions were updated in `v1.0.3` to be more optimized and better for performance and execution time.
- Reworked diagnostics to collect errors across the entire recovered file instead of stopping after parsing or the first failed top-level item. Output is deduplicated and grouped in `syntax`, `reference`, `type`, `runtime`, then `internal` order for both CLI execution modes and WASM.
- Fixed function capture initialization in both CLI execution modes so lexical names and built-ins such as `@println` remain available inside functions, while recursive and forward function references still resolve correctly.
- Fixed bytecode jump patching for optimized identifier conditions. Loops could previously jump past the rest of a function and turn an explicit array return into `void`, causing false assignment type errors such as `a = stabilize => |c, SIZE|` in `stress_test.zk`.
- Rebuilt the website WASM assets with the corrected function environment and added cache-busting for both the generated JavaScript module and WASM binary so the demo cannot keep executing stale runtime code after a rebuild.
- Added 3 new tests for assigning values from funtion returns (`function_assign.zk`), logical NOT flipping boolean values (`logical_not.zk`) and logical OR for `if condition || other_condition {...}` (`logical_or.zk`)
- Made the error handler log all current errors for a file in order of type (syntax, reference, type, runtime, internal if found) instead of prioritizing syntax errors over anything else and not logging any other type until all current syntax errors are fixed
- Fixed the parser silently accepting semicolons after expression statements. Calls such as `@println => |"Hello"|;` now report the unexpected semicolon, recover at that token, and continue collecting later errors without executing the malformed line.
- Fixed a regression where zero-argument calls using `=> ||` were parsed as logical OR when followed by comparisons, another statement, or the end of a line. Empty-call pipes are now resolved from call context while `||` remains a logical operator in ordinary `if`, `while`, `for`, and value expressions. Exact duplicate diagnostics are collapsed without hiding distinct errors at the same source location.
